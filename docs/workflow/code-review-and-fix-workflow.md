# 代码审查与修复标准工作流程

本文档沉淀了一套经过实践验证的代码审查→修复→测试→提交完整工作流程，
适用于针对已有代码库的质量改进、性能优化和 Bug 修复场景。

---

## 工作流程总览

```
代码审查（code-reviewer）
       ↓
   制定修复方案
       ↓
方案评审（architecture-designer）← 最多 3 轮讨论
       ↓
   执行代码修改
       ↓
创建/更新测试（general-purpose agent）
       ↓
   运行测试（100% 通过为门槛）
       ↓
最终代码评审（code-reviewer）
       ↓
  评审通过后提交（/commit）
       ↓
   推送到远端（git push）
       ↓
沉淀工作流程文档（本文档）
```

---

## 阶段一：代码审查

### 目标
全面扫描代码库，识别性能问题、状态机闭环缺陷、并发安全问题和边界场景遗漏。

### 执行方式
使用 `code-reviewer` 子 agent，明确指定审查维度：
- **性能分析**：热路径、锁竞争、不必要的内存分配、重复存储查询
- **状态机闭环**：状态转换是否完整、是否有死锁状态或状态泄漏
- **边界问题**：空集合、并发竞态、版本冲突、存储后端差异

### 输出要求
- 每个问题附上具体代码位置（`file_path:line_number`）
- 按优先级分类（Critical / Important / 补充）
- 说明问题的影响范围和复现条件

---

## 阶段二：制定修复方案

### 原则
- 先深入阅读相关代码文件，基于实际代码现状制定方案
- 不基于假设或猜测制定方案
- 每个修复方案需说明：问题根因、修复思路、预期效果

### 方案制定后的自检
- 方案是否引入新的并发问题？
- 是否保持接口向后兼容？
- 是否需要同步修改测试？

---

## 阶段三：方案评审（最多 3 轮）

### 目标
通过 `architecture-designer` 子 agent 对修复方案进行技术可行性评审，
确保方案正确、无遗漏风险。

### 评审内容
- 技术可行性：方案逻辑是否正确
- 完整性：是否覆盖所有边界情况
- 是否有更优雅的替代方案
- 优先级排序建议

### 迭代机制
- 评审后按意见调整方案
- 最多经过 3 轮讨论达成一致
- 达成一致后才能进入实施阶段

### 注意事项
- 评审意见需逐条确认，不能只选择性采纳
- 对于"通过"的方案，若后续发现新问题仍需修改

---

## 阶段四：执行代码修改

### 顺序建议
按优先级顺序执行：P0 → P1 → P2，优先修复影响正确性的问题。

### 修改规范
- 每次修改一个独立问题，不混入不相关改动
- 修改后立即运行 `go build ./...` 验证编译
- 保持锁顺序一致（参考现有代码的锁使用约定）

### 常见陷阱
- **锁顺序**：多个 mutex 的获取顺序必须全局一致，否则导致死锁
- **乐观锁重试**：版本冲突时需要重新 GetAccount 拿最新版本，不能直接递增版本号
- **终态保护**：熔断/禁用等严重状态不应被轻量状态覆盖
- **全量写 vs 原子更新**：高并发场景下，`SaveUsages`（全量覆写）应替换为 `CalibrateRule`（原子更新）

---

## 阶段五：创建/更新测试用例

### 执行方式
使用 `general-purpose` 子 agent，提供完整的修改内容描述和测试要求。

### 测试覆盖要求
- 每个修复点至少有 1 个正向测试 + 1 个边界测试
- 并发安全修复必须附带 `race detector` 验证（`go test -race`）
- 不修改业务代码，只创建/更新测试文件
- 对于需要真实外部服务（如 Redis）的测试，使用 mock 或 integration build tag 跳过

### 测试质量检查
- 断言是否有实际意义（不能只断言"不报错"）
- mock 的调用次数验证是否符合优化预期
- 并发测试的 goroutine 数量是否足以触发竞态

---

## 阶段六：运行测试

### 命令
```bash
go test ./... -race -count=1
```

- `-race`：启用 race detector
- `-count=1`：禁用测试缓存，确保每次都实际运行

### 门槛
**100% 通过**，有任何失败都必须先修复，不能跳过。

---

## 阶段七：最终代码评审

### 执行方式
使用 `code-reviewer` 子 agent，明确告知这是第几轮评审以及上一轮的修复情况。

### 评审重点
- 上一轮问题是否都已修复
- 修复逻辑是否正确（无新引入的 bug）
- 注释是否与代码一致（特别是 ARGV 格式等协议描述）
- 死代码、错别字等清洁度问题

### 通过标准
明确给出 **Pass** 结论才能进入提交阶段。

---

## 阶段八：代码提交与推送

### 提交规范
使用 `/commit` 命令，提交信息格式：
```
fix: <简短描述>

<详细说明每个修复点，包括问题根因和修复方式>

新增测试：<新增测试用例数量>个，覆盖所有修复点
```

### 推送
```bash
git push origin <branch>
```

确认推送成功（查看 remote 输出）。

## 常用 Agent 对应关系

| 阶段 | Agent | 说明 |
|------|-------|------|
| 代码审查 | `code-reviewer` | 全量或 diff 审查，识别问题 |
| 方案评审 | `architecture-designer` | 技术可行性和架构合理性 |
| 测试编写 | `general-purpose` | 创建/更新测试用例 |
| 最终 review | `code-reviewer` | 验证修复内容的正确性 |
| 探索代码库 | `Explore` | 快速理解代码结构和调用关系 |

## 本次工作流程的实践记录

**项目**：lumin-acpool（账号池负载均衡器）  
**背景**：针对代码审查发现的 11 个问题进行修复

**工作流程执行情况**：
- 代码审查：1 轮，发现 4 个 Critical + 7 个 Important 问题
- 方案评审：1 轮，评审后调整了 5 个方案（Fix-5/7/8/9 需修改，Fix-12 确认不修改）
- 代码修改：覆盖所有 11 个修复点，2 轮编译验证通过
- 测试：新增 63 个测试用例，100% 通过
- 最终 code review：2 轮（第一轮 Fail，修复 3 个问题后第二轮 Pass）
- 提交推送：单次成功

**经验教训**：
1. 方案评审阶段要重点关注评审意见中"需修改"的方案，不能只看"通过"
2. Lua 脚本的注释必须与实际 ARGV 布局一致，否则会严重误导维护者
3. 乐观锁重试保护需要覆盖所有"更严重"的状态，不只是最终态（Banned 等）
4. `SaveUsages` 全量覆写是高并发场景中的常见陷阱，应优先使用原子操作接口
5. 并发安全修复（数据竞争）应配合 `-race` 标志的测试来验证

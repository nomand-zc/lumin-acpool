# 编码规范

## 命名规范

| 类别 | 规范 | 示例 |
|------|------|------|
| 包名 | 小写单词 | `balancer`, `selector` |
| 接口 | 行为命名，无 I 前缀 | `Balancer`, `Selector` |
| 实现 | 小写私有 + `default` 前缀 | `defaultBalancer` |
| 常量 | CamelCase | `StatusAvailable`, `UpdateFieldStatus` |
| 文件名 | 小写下划线 | `default_balancer.go` |

## 架构原则

- 遵循 SOLID、GRASP 和 YAGNI 原则
- 接口约束高于实现约束
- 配置使用 `With*` Functional Options，新增 Option 不修改已有签名
- `Account` 只持数据，`Balancer` 只编排流程，算法封装在 `Selector`
- 非必要不导出

# 测试规范完整版（单元测试 + 集成测试 + 基准测试）

## 一、单元测试规范（完整版）

### 1. 基础命名与文件规范

1. 测试文件必须以 `_test.go` 结尾，与被测文件**同目录、同包**

2. 测试函数命名：`TestXxx(t *testing.T)`，`Xxx` 为被测函数/方法名，首字母大写

3. 子测试命名：`t.Run("测试场景描述", func(t *testing.T){})`，场景名语义化（如：正常入参、空参数、参数非法）

### 2. 依赖与工具规范

1. 强制使用 Go 标准 `testing` 包，**禁止使用 testify/assert**，原生判断：

    ```Go

    if got != want {
        t.Errorf("got %v, want %v", got, want)
    }
    ```

2. 外部依赖 Mock 规则：

   - MySQL：使用 `sqlmock`

   - Redis：使用 `miniredis`

   - HTTP接口：使用 `net/http/httptest`

   - 禁止跳过外部依赖测试、禁止硬编码绕过依赖

### 3. 用例编写规范（Table-Driver 强制）

1. 所有逻辑测试**必须使用表驱动模式**，定义用例结构体切片

2. 用例结构体包含字段：`name`(场景名)、`args`(入参)、`want`(预期值)、`wantErr`(是否预期错误)

3. 用例覆盖范围：

   - 正常业务流程

   - 边界值（空值、0、最大值、最小值）

   - 非法参数（类型错误、格式错误、空指针）

   - 异常分支（数据库报错、依赖超时、返回空数据）

### 4. 并发与安全规范

1. 涉及并发、共享内存、锁的测试，**必须添加 ****`-race`**** 检测**

2. 测试函数内部禁止跨用例共享变量，所有变量独立初始化

3. 禁止在单元测试中启动真实外部服务（MySQL/Redis/HTTP）

### 5. 断言与日志规范

1. 失败时必须打印**清晰的对比信息**，禁止仅返回 `t.Fail()`

2. 禁止使用 `fmt.Print`，测试日志使用 `t.Logf/t.Errorf`

3. 一旦发现错误，使用 `t.Fatalf` 终止当前子测试，避免无效后续执行

### 6. 覆盖率要求

1. 核心业务逻辑单元测试覆盖率 ≥ 90%

2. 工具类、通用组件覆盖率 ≥ 90%

3. 执行命令：`go test -cover -coverprofile=cover.out && go tool cover -html=cover.out`

## 二、集成测试规范（完整版）

### 1. 基础定义

集成测试用于验证**模块间交互、接口联调、配置加载、真实依赖连通性**，区别于单元测试（单元测试隔离依赖，集成测试使用真实/近似真实环境）。

### 2. 文件与命名规范

1. 测试文件：以 `_integration_test.go` 结尾，与被测文件同包

2. 测试函数：`IntegrationTestXxx(t *testing.T)`

3. 构建标签：文件首行添加 `//go:build integration`，避免和单元测试一起执行

### 3. 环境与依赖规范

1. 允许使用**测试环境的真实中间件**（专用测试库、测试队列、测试缓存）

2. 禁止直接操作生产/预发环境数据

3. 数据隔离：测试前初始化数据，测试后**自动清理测试数据**（tearDown）

4. 配置文件：使用独立的集成测试配置（`config.integration.yaml`）

5. 存储层的依赖可以使用docker来启动，比如redis、mysql

### 4. 测试范围规范

1. 必须覆盖：

   - 接口入参/出参格式校验

   - 数据库事务、联表查询、索引有效性

   - 缓存与数据库一致性逻辑

   - 消息生产/消费链路

   - 配置文件加载、环境变量读取

2. 禁止覆盖：纯函数逻辑、无需联调的工具方法（交由单元测试覆盖）

### 5. 执行与隔离规范

1. 执行命令：`go test -tags=integration -v ./...`

2. 测试用例之间**完全隔离**，不依赖其他用例执行结果

3. 支持批量执行与CI/CD集成，无人工干预

### 6. 失败处理规范

1. 集成测试失败优先排查：环境连通性、数据初始化、依赖服务状态

2. 测试失败必须保留现场日志，禁止静默失败

## 三、基准测试规范（完整版）

### 1. 基础命名与文件规范

1. 基准测试文件：以 `_benchmark_test.go` 结尾，与被测文件同包

2. 基准函数命名：`BenchmarkXxx(b *testing.B)`，`Xxx` 为被测函数名

3. 子基准测试：`b.Run("场景名", func(b *testing.B){})`

### 2. 强制覆盖范围

1. Pick 流程全链路函数、核心算法、数据处理函数

2. 高频调用接口、数据库查询、缓存操作

3. 并发处理、序列化/反序列化、切片/Map操作

### 3. 编写核心规范

1. 循环体内只执行**被测逻辑**，禁止包含初始化、日志、断言等耗时操作

2. 固定写法：

    ```Go

    func BenchmarkXxx(b *testing.B) {
        // 初始化代码放在 b.ResetTimer() 之前
        setup()
        b.ResetTimer()
        // 被测逻辑放在循环内
        for i := 0; i < b.N; i++ {
            Xxx() // 被测函数
        }
    }
    ```

3. 并发基准测试：使用 `b.RunParallel` 测试高并发场景性能

### 4. 依赖与Mock规范

1. 涉及网络/存储调用：优先使用docker环境来处理依赖的网络服务，如果docker解决不了的，允许使用mock

### 5. 执行与指标规范

1. 执行命令：`go test -bench=. -benchmem -count=5`

2. 必须关注指标：

   - 每次操作耗时（ns/op）

   - 每次操作内存分配（B/op）

   - 每次操作分配次数（allocs/op）

3. 每次版本迭代，核心函数基准测试**性能不得下降**

### 6. 禁忌规范

1. 禁止在基准测试循环内使用 `t.Log`、`fmt` 等输出

2. 禁止循环内包含无关逻辑，导致测试结果失真

3. 禁止不设置计时器重置，导致初始化时间计入性能结果

---

## 四、通用测试禁忌（全类型测试通用）

1. 禁止测试用例执行顺序依赖

2. 禁止测试代码中包含业务逻辑

3. 禁止忽略错误返回、不做异常断言

4. 禁止提交无法通过的测试代码

5. 禁止Mock不完整、依赖未覆盖就提交代码

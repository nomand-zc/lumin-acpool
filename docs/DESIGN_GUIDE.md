# 需求方案设计指南

**适用场景**：当 agent 需要为一个新功能需求输出可直接用于实现的设计方案文档时，遵循本指南。  
**核心目标**：方案文档输出后，负责实现的 agent 能无歧义地执行，不需要猜测、不需要追问，实现结果与预期严格一致。

---

## 一、设计前的强制探索步骤

在写任何一行方案内容之前，必须先完成以下探索，否则方案必然产生遗漏或冲突。

### 1.1 阅读项目规范文档

| 文档 | 必读内容 |
|------|---------|
| `ARCHITECTURE.md` | 设计约束、模块职责边界、禁止事项 |
| `docs/CONVENTIONS.md` | 命名规范、架构原则（接口命名、Functional Options、导出规则等） |
| `docs/TESTING.md` | 测试规范（Table-Driven、禁止 testify、mock 策略、文件命名规则） |
| `docs/CODE_REVIEW.md` | 代码审查标准 |

**不阅读规范就设计，等于让实现者在不知道游戏规则的情况下写代码。**

### 1.2 深度探索现有代码库

设计前必须回答以下问题：

**接口与数据结构**
- 需求涉及的数据结构在哪个包中定义？与哪些现有类型有关联？
- 需要新增的接口是否在现有接口中已有部分实现？（避免重复定义）
- 现有接口的签名是什么？（避免方案中的接口与现有签名冲突）

**依赖关系**
- 新模块依赖哪些现有包？这些依赖方向是否会产生循环？
- 现有包是否已经依赖了新模块所在的包？（循环依赖检查）
- 如果 `storage` 包需要使用新模块的类型，谁应该持有类型定义？

**现有实现细节**
- 类似功能在其他模块中是如何实现的？（参照 Pattern，避免风格不一致）
- 现有代码中是否有与新需求重叠的逻辑？（考虑复用还是独立维护）
- 存储层已有哪些后端实现？新接口需要在哪些后端同步实现？

**副作用与兼容性**
- 修改现有接口（如 `storage.Storage` 聚合接口）会影响哪些调用方？
- 是否存在需要迁移的现有类型定义？（如 `health.LeaderElector` → `election.LeaderElector`）

## 二、方案文档的必要章节

一份合格的方案文档必须包含以下所有章节，缺少任何一章都会导致实现时产生歧义。

### 章节清单

| 章节 | 作用 | 缺失后果 |
|------|------|---------|
| 背景与目标 | 明确「为什么做」和「做到什么程度」 | 实现者不理解边界，容易过度实现或实现不足 |
| 架构概览（模块位置 + 数据流） | 明确新代码放在哪里、数据怎么流动 | 实现者不知道在哪里加文件 |
| 数据结构定义（含注释） | 所有类型的字段、语义、约束 | 实现者自行猜测字段含义，产生语义偏差 |
| 接口定义（含方法注释） | 所有接口方法的签名、参数语义、返回值约定 | 实现者自行决定接口，可能与调用方不匹配 |
| 核心计算/业务逻辑 | 算法伪代码、边界条件、错误处理策略 | 实现者自行决定逻辑，产生与预期不符的结果 |
| 配置选项（Options） | With* 函数列表、必填/可选、默认值 | 实现者不知道哪些必填，漏掉校验逻辑 |
| 多实例/并发策略 | 分布式场景下的行为定义 | 集群部署时出现数据不一致或重复执行 |
| 包结构与文件规划 | 每个新文件的职责和位置 | 实现者随意创建文件，结构混乱 |
| 验收标准（AC） | 每个功能点的具体测试场景 | 测试写错或遗漏，代码质量无法保障 |
| 不在范围内 | 明确排除什么 | 实现者误实现了超出需求的功能 |

## 三、数据结构定义的写法规范

数据结构是方案文档中最容易产生歧义的部分，必须做到以下几点。

### 3.1 每个字段必须有统计口径说明

仅给出字段名和类型是不够的。必须说明：
- 统计范围（全量账号？只统计 Available？还是某个子集？）
- 数据来源（哪个 Store 接口？哪个方法？）
- 边界值处理（为 0 时返回什么？为负数时如何钳制？）
- 与其他字段的关联约束（如「各字段之和 == TotalAccounts」）

**示例（错误写法）**：
```go
// EffectiveAvailable 可用账号数
EffectiveAvailable int
```

**示例（正确写法）**：
```go
// EffectiveAvailable 当前真正可被 Pick 流程选出的账号数。
// 统计范围：在 TotalAccounts 基础上经过双重过滤：
//   ① Status == Available
//   ② 所有 UsageRule 对应的 TrackedUsage.IsThrottled(safetyRatio) 均返回 false
//      IsThrottled 内部处理 rule.Total <= 0（视为无限额，不排除账号）
// 说明：并发占用在 OccupancyTotal 中单独统计暴露，不参与此过滤。
EffectiveAvailable int
```

### 3.2 明确哪些判断逻辑属于业务层而非本模块

凡是需要根据阈值做「健康/不健康」「需要/不需要」判断的逻辑，统一由业务层处理。本模块只存储原始指标值。

**错误设计**：`NeedMoreAccounts bool` 放入存储结构  
**正确设计**：存储 `EffectiveAvailable int`，业务层自行判断是否需要补充

### 3.3 跨模块共享类型的归属原则

当一个类型需要同时被 A 包和 B 包使用时，必须在设计阶段明确它放在哪个包，而不是等实现时再决定。

判断规则：
- 如果类型是某个领域实体的自然属性 → 放在领域包（如 `account`）
- 如果类型是某个基础设施的内部概念 → 放在基础设施包（如 `storage`）
- 如果放在任何一方都会造成循环依赖 → 下沉到被双方依赖的低层包

**本项目案例**：`Metrics` 类型被 `storage` 包（`MetricStore` 接口参数）和 `observer` 包同时使用，若放在 `observer` 则 `storage → observer` 形成循环，正确做法是放在 `account` 包。

## 四、接口设计的写法规范

### 4.1 必须给出完整的 Go 签名

方案文档中的接口必须是可以直接复制到代码文件中的合法 Go 代码，包含：
- 完整的方法签名（含 `ctx context.Context`）
- 返回值类型的完整包路径（如 `*account.Metrics` 而非 `*Metrics`）
- 每个方法的 godoc 注释，说明成功返回、错误返回（如 `ErrNotFound`）

### 4.2 批量接口的必要性需要在设计阶段评估

如果某个操作在实现时必然需要对 N 个对象逐一调用，设计时就要决定：
- 是增加批量接口（如 `GetCurrentUsagesBatch`）
- 还是接受 N 次调用（并说明理由和性能预期）

**不能留给实现者自行决定**，否则不同实现者可能做出不同选择，导致性能差异或接口不统一。

### 4.3 存储语义必须明确

每个写入接口必须说明：
- **upsert**（覆盖写，同 key 只保留最新）
- **append**（追加，同 key 可有多条）
- **insert**（幂等插入，已存在时报错）

每个读取接口必须说明：
- key 不存在时的行为（返回 `ErrNotFound`？返回零值？）
- 空集合时的行为（返回 `nil`？返回空切片？）

## 五、核心逻辑的写法规范

### 5.1 用伪代码描述执行顺序，而非文字描述

文字描述容易模糊，伪代码有明确的执行顺序语义。

**错误写法**：
> 对每个 Provider 计算指标，然后汇总到全局维度。

**正确写法**：
```
1. SearchProviders(nil) → 失败时返回 error，终止
2. 初始化 globalMetrics（全零值）
3. 对每个 Provider:
   a. collectMetrics(providerType, providerName)
   b. 若失败 → continue，不更新快照，不累加到 globalMetrics
   c. 若成功 → SaveMetrics(providerMetrics)，将各字段累加到 globalMetrics
4. SaveMetrics(globalMetrics) → 失败时返回 error
```

### 5.2 错误处理策略必须逐一说明

每个可能失败的操作，都需要明确：
- 失败时的行为（跳过本条？终止整个流程？返回 error？只 log？）
- 失败时对后续操作的影响（是否影响其他 Provider 的处理？是否影响全局维度？）
- 错误值是否透传给调用方

**典型问题**（来自本次设计过程）：  
`RefreshNow` 在单个 Provider 失败时是否返回 error？是否继续处理其余 Provider？失败的 Provider 是否累加到全局？这三个问题如果不在方案中明确，实现者会做出三种不同的选择。

### 5.3 边界条件必须枚举

对于数值计算，必须说明以下边界的处理方式：
- 分母为 0（如 `rule.Total == 0` 时的除法）
- 结果为负数（如 `EstimatedRemain < 0` 时是否钳制为 0）
- 输入集合为空（如没有账号时 `SuccessRate` 返回什么）
- 批量查询结果中某个 key 不存在（视为空还是报错）

## 六、依赖复用的决策规范

当新模块的逻辑与现有模块存在相似代码时，必须在方案设计阶段做出明确决策，不能留给实现者自行判断。

### 6.1 决策框架

| 情况 | 建议策略 |
|------|---------|
| 完全相同的公式/算法，且语义一致 | 下沉到公共类型的方法（如 `TrackedUsage.IsThrottled()`） |
| 逻辑相似，但数据来源/调用模式不同 | 各模块独立实现，不强行复用 |
| 跨模块注入共享组件（依赖注入） | 谨慎评估——会模糊模块边界，仅在明确有收益时使用 |
| 存在公共接口概念（如 `LeaderElector`） | 提取到独立包（如 `election/`），多模块共同 import |

### 6.2 何时不应该复用

- 两个模块的调用模式不同（实时 vs. 批量）
- 两个模块的副作用不同（有 callback vs. 只读统计）
- 强行复用会导致一个模块依赖另一个模块（形成不合理的依赖方向）

### 6.3 方案文档中的表述方式

决定复用时，必须在方案中明确：
1. 复用的具体内容（哪个方法/类型）
2. 对被复用方的改动（新增方法、修改签名等）
3. 哪些调用方需要同步修改

## 七、验收标准（AC）的写法规范

验收标准是连接「设计意图」和「测试实现」的桥梁。写得模糊等于没写。

### 7.1 每条 AC 的结构

| 字段 | 要求 |
|------|------|
| 编号 | 层次化编号（AC-1.1、AC-1.2），便于引用 |
| 验收条件 | 一句话描述「什么情况下通过」 |
| 测试用例场景 | 具体的输入、操作、预期输出 |

**错误写法**：
> AC-1.1：EffectiveAvailable 计算正确

**正确写法**：
> AC-1.1：`TrackedUsage.IsThrottled(0.95)` 在 `EstimatedUsed()=950, Total=1000` 时返回 true，账号被排除；`Total=0` 时返回 false，账号不被排除

### 7.2 必须覆盖的测试维度

| 测试类型 | 必须包含的场景 |
|---------|--------------|
| 正常路径 | 典型输入的预期输出 |
| 边界条件 | 零值、空集合、分母为零、负数钳制 |
| 错误路径 | 存储层返回 error 时的行为 |
| 并发安全 | 并发读写、data race 检测（`-race`） |
| 生命周期 | Start/Stop 重复调用、goroutine 退出 |
| 多实例 | 共享存储时的一致性行为（集成测试） |
| 性能基线 | 关键路径的 ns/op 上限（基准测试） |

### 7.3 指定测试文件和执行命令

每组 AC 必须明确：
- 测试文件路径（如 `observer/observer_test.go`）
- 函数名前缀（如 `TestCollector_EffectiveAvailable`）
- 执行命令（如 `go test -race ./observer/...`）

## 八、多实例与并发策略的必要说明

凡是涉及后台 goroutine 或跨实例共享状态的功能，必须在方案中说明以下场景：

| 场景 | 必须说明的内容 |
|------|--------------|
| 单机部署 | 默认行为（无需注入任何组件） |
| 集群 + 共享存储 | 如何防止多实例重复执行（分布式锁、LeaderElector） |
| 集群 + 非共享存储 | 是否每实例独立计算（Memory 后端不跨进程） |
| leader 宕机 | IsLeader 不可用时的 fallback 行为 |
| 并发写入 | 存储语义（upsert last-write-wins？还是需要版本控制？） |

## 九、「不在本次范围内」章节的重要性

这是最容易被忽略、但对避免过度实现至关重要的章节。

必须明确排除以下类型的内容：
- **判断逻辑**：本模块只提供原始指标，健康/告警判断由业务层负责
- **历史时序**：只存最新快照，历史趋势不在本模块
- **网络服务**：本模块是 SDK 库，不提供 HTTP endpoint
- **下游触发**：指标变化时不主动推送或告警
- **未来可能需要的功能**：明确标注为「YAGNI，当前不实现」

以下是从 `observer` 模块设计过程中提炼出的核心教训，每条对应一类「因为方案不完整导致实现时需要重新讨论」的问题：

| N+1 查询问题（`GetCurrentUsages` 逐个调用） | 设计时未评估批量接口必要性 | 第四章 4.2 |
| RefreshNow 错误语义不清（单 Provider 失败是否返回 error） | 核心逻辑的错误处理未逐一说明 | 第五章 5.2 |
| SafetyRatio 计算与现有代码不对齐（`Total<=0` 未处理） | 设计前未阅读现有代码实现细节 | 第一章 1.2 |

| mock 策略未说明（手写 mock vs memory store） | 验收标准未引用测试规范 | 第一章 1.1 + 第七章 |
| `PoolHealth` 字段未经讨论放入 `Metrics` | 未确认「健康判断」属于业务层 | 第三章 3.2 + 第九章 |

## 十一、输出格式要求

方案文档必须以 Markdown 格式输出，存放在 `docs/superpowers/specs/` 目录下，文件名格式为 `{YYYY-MM-DD}-{feature-name}-design.md`。

文档内的代码片段必须是合法的 Go 代码（可直接用于实现），不使用伪类型或占位符。

文档状态字段：
```
**日期**：YYYY-MM-DD
**状态**：待实现 / 实现中 / 已完成
**关联需求**：{需求来源引用}
```

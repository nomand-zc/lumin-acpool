# 文档编写规范 (DOC_CONVENTIONS)

本文档是 lumin-acpool **文档知识库的内容权威来源**（SSOT），定义"写什么"与"怎么写"。

> **与其他规范的关系**
> - `DOC_SYNC.md`：引用本文档的内容约束，自身聚焦漂移检测的**触发时机与流程**
> - `.codebuddy/agents/doc-maintainer.md`：执行文档更新时遵循本文档的所有规则

---

## 0. 核心指导思想

### 0.1 文档存在的目的：为 agent 提供完整认知闭环

本项目的文档知识库首要服务对象是 **AI agent**，其次才是人类开发者。

Agent 在执行任务时，失败的根源往往不是能力不足，而是**上下文质量不足**——它不知道系统为什么这样设计、不清楚团队的边界约束、无法验证自己的输出是否正确、也得不到改进的信号。

Harness Engineering 将这四种认知需求对应为四类文档职能，构成一个完整的认知闭环：

```
inform（理解背景与意图）
  → constrain（明确边界与约束）
    → verify（校验输出是否正确）
      → feedback（收到改进信号）
            ↑__________________________|
```

**维护这四类文档的目的**：让 agent（以及人类）在任意任务中都能做出与团队意图一致的决策，并能自我校验和持续改进。

这个目的决定了文档的写作标准：**不是"内容完整"，而是"agent 拿到后能做出正确决策"**。

### 0.2 文档不是越多越好：信噪比优先

文档的价值不在于体量，而在于**信息增量**。冗余内容不是中性的——它稀释真正有用的信息，增加 agent 和人类的认知负担，并制造额外的漂移风险（内容越多，与代码不同步的概率越高）。

以下三类内容**显而易见，不应写入文档**：

| 类型 | 原因 | 示例 |
|------|------|------|
| 代码本身已能表达的内容 | 读代码比读文档更准确，重复只带来漂移风险 | 函数签名、字段类型、方法返回值 |
| 行业或语言的通用知识 | 不是项目特有的决策，属于背景常识 | Go 接口惯例、HTTP 状态码含义 |
| 可由已有内容直接推论的结论 | 没有增量信息，只是增加冗余 | A 和 B 都在文档里，C 只是 A+B 的推论 |

**写入前的核心判断**：

> **"如果没有这条内容，读者或 agent 会做出错误决策吗？"**
>
> - 不会 → 不写
> - 会 → 才值得写入

这条判断标准优先于所有具体规则。当具体规则与它冲突时，以它为准。

## 1. 文档职能分类

遵循 Harness Engineering 的四类职能框架。**每篇文档必须在开头声明其职能类型**。

| 职能 | 核心问题 | 本项目示例 |
|------|---------|-----------|
| **inform** | 系统是什么？为什么这样设计？ | `ARCHITECTURE.md`, `design-docs/*.md` |
| **constrain** | 必须怎么做？禁止做什么？ | `CONVENTIONS.md`, `DOC_CONVENTIONS.md` |
| **verify** | 如何证明它是正确的？ | `COMMIT_ACCEPTANCE.md`, `ENVIRONMENT.md` |
| **feedback** | 如何改进质量？ | `CODE_REVIEW.md`, `DOC_SYNC.md` |

### 1.1 新建文档决策树

```
需要新建文档？
  ├─ 已有文档能否容纳？ → 能 → 更新现有文档，不新建
  └─ 必须新建？
       ├─ 解释设计决策、跨模块流程    → inform 类
       ├─ 强制规则、禁止项、命名约定   → constrain 类
       ├─ 验收标准、基准值、环境配置   → verify 类
       └─ 审查 checklist、改进反馈流程 → feedback 类
```

一篇文档**只能承担一种主职能**，最多兼含相邻职能（如 `constrain + verify`）。若发现需要兼含三种以上职能，说明文档边界设计有误，应拆分。

## 2. 准入原则：按职能写对的内容

### 2.1 各职能类型的内容要求

四类文档各有其合法内容范围与特殊要求：

#### inform（描述型）

**写什么**：
- 架构决策：代码无法表达的 why，设计取舍与背景
- 接口约束：调用方必须知道的契约、状态机规则、不变量
- 流程全景：跨模块协作链路，读代码无法一目了然的

**特殊要求**：
- 以"why"为核心，不重复代码已能表达的"what"和"how"
- 状态机、流程图等需与代码实现严格对齐，是漂移高发区
- 禁止写入字段类型、函数签名等可直接读代码获取的细节

#### constrain（约束型）

**写什么**：
- 强制性命名规范、架构原则、编码惯例
- 明确的禁止项（禁止做什么、禁止用什么）
- 必须满足的结构性要求

**特殊要求**：
- 每条规则必须**可执行、可验证**——模糊的"建议"不属于 constrain 类
- 禁止项需说明违反后果或检测方式，不能只列"禁止 XX"
- 规则应覆盖正反两面：允许的 + 禁止的

#### verify（验证型）

**写什么**：
- 可量化的验收标准：覆盖率阈值、性能基线、响应时间上限
- 环境依赖清单：工具版本、服务配置、启动命令
- 门禁配置说明：hook 列表、触发条件、跳过方式

**特殊要求**：
- 所有数值必须有来源（对应配置文件或代码中的常量）
- 基线值随代码变更需同步更新，否则立即产生漂移
- 禁止写"大约"、"通常"等模糊量词，verify 类文档要求精确

#### feedback（反馈型）

**写什么**：
- 审查 checklist：分级的检查项（Critical / Important / Minor）
- 改进流程：门禁顺序、报告格式、修复路径
- Agent 行为规范：执行步骤、边界条件、输出格式

**特殊要求**：
- checklist 项必须**可判断真假**，不能是主观判断题
- 流程描述需与实际脚本/hook 配置保持一致，是漂移高发区
- Agent 相关规范需明确"禁止"和"必须"，不能只写"建议"

---

跨越多个职能的内容，写入**主职能**对应的文档，其他文档通过链接引用。

### 2.2 禁止写入

- **代码可直接表达的内容**：字段类型、函数签名、方法返回值——读代码更准确
- **无增量信息的换一种说法**：已有描述的同义重写
- **多文档间的重复块**：同一段内容出现在两处——用链接引用替代，DRY 原则适用于文档间
- **假设性内容**："可能"、"将来考虑"、"预计支持"
- **纯实现细节**：算法内部逻辑、变量命名理由、临时性代码注释的翻版
- **超过 3 层嵌套的列表**

### 2.3 边界判断：读代码的替代成本

当不确定某内容是否值得写入文档时，问自己：

> **"读代码能否在 5 分钟内得到同等信息？"**
>
> - 能 → 不写，代码是更可靠的真相来源
> - 不能（跨多个文件、有历史决策背景、理解需要领域知识）→ 才值得写入文档

## 3. 结构与格式规范

### 3.1 文档头部必须包含

```markdown
# 文档标题（名词短语，体现职能）

一句话说明本文档**是什么**、**为谁服务**、**不包含什么**。

> **与其他文档的关系**（仅当存在直接依赖关系时写）
> - `相关文档.md`：用一句话说明关系（依赖 / 被依赖 / 补充）
```

### 3.2 标题层级

- 最多使用 `##`（二级）到 `####`（四级）
- `#` 只用于文档标题，全文唯一
- 禁止跳级（`##` 下直接用 `####`）

### 3.3 表格 vs. 列表的选择

| 场景 | 推荐形式 |
|------|---------|
| 有两个以上维度需要对比（如：类型、描述、示例） | 表格 |
| 枚举同质项目，无需对比 | 无序列表 |
| 有明确执行顺序的步骤 | 有序列表 |
| 键值映射（如：字段名 → 说明） | 表格 |

### 3.4 代码块使用规则

- 只放**可直接运行或复制使用**的内容：命令、配置片段、接口签名
- 禁止放伪代码或"示意性"代码（用文字描述代替）
- 必须标注语言标识符：` ```go `, ` ```bash `, ` ```sql `

### 3.5 图形（ASCII / Mermaid）

图形**使用门槛高**，满足以下全部条件才允许引入：

1. 表达的信息用文字或表格无法清晰呈现
2. 图形内容能与代码同步维护（否则图形会成为漂移源）
3. 图形改动需在报告中标注 `DIAGRAM_UPDATED` 并附说明（见第 4.3 节）

## 4. 修改质量门槛

### 4.1 修改前三问（全部满足才允许修改）

1. 代码变更导致现有文档**事实错误**或**流程漂移**
2. 该信息是**读代码无法替代**的（必须在文档中存在）
3. 修改后文档**等长或更短**（有等量删除才允许内容增长）

> 第 3 条是关键约束：文档膨胀是质量下降的主要原因。增加新内容必须同步删除已被代码取代或冗余的旧内容。

### 4.2 漂移 vs. 非漂移

| 情况 | 是否漂移 | 是否需要更新文档 |
|------|---------|----------------|
| 接口签名变更，文档描述的是旧签名 | **是** | 是 |
| 纯重构，代码结构变化但行为不变 | **否** | 否 |
| 新增枚举值，文档未收录 | **是** | 是 |
| 新增测试用例 | **否** | 否 |
| 性能优化，不影响调用契约 | **否** | 否 |
| 变量重命名，不影响接口 | **否** | 否 |

### 4.3 复杂修改的执行原则

复杂修改（diff 行数多、涉及架构图、大量删除）无需切换建议模式，直接执行。但必须在执行前额外验证：

- **修改前三问全部通过**（§ 4.1）
- **逐条核查**：每一处修改单独满足"没有它会做出错误决策"的写入标准（§ 0.2）
- **等量删除**：增加内容必须有对应删除，不允许净增长

架构图（ASCII / Mermaid）除外——图形改动需在报告中标注 `DIAGRAM_UPDATED` 并附说明，便于人工复核。

## 5. 文档间引用规范

### 5.1 引用而非复制

当两篇文档需要涉及同一规则时，**指定唯一的权威来源**，其他文档用 Markdown 链接引用：

```markdown
> 内容约束的完整规则见 [DOC_CONVENTIONS.md § 2](DOC_CONVENTIONS.md#2-准入原则按职能写对的内容)。
```

禁止将同一段规则复制到多处，即使措辞略有不同。

### 5.2 SSOT 映射（各类规范的权威来源）

| 内容类型 | 权威来源 |
|---------|---------|
| 文档内容创作规则 | `docs/DOC_CONVENTIONS.md`（本文档） |
| 文档漂移检测与同步流程 | `docs/DOC_SYNC.md` |
| 代码编写规范、命名规则 | `docs/CONVENTIONS.md` |
| 测试规范（单元/集成/基准） | `docs/TESTING.md` |
| 代码审查 checklist | `docs/CODE_REVIEW.md` |
| 验收标准与覆盖率阈值 | `docs/COMMIT_ACCEPTANCE.md` |
| 系统架构与组件关系 | `ARCHITECTURE.md` |

### 5.3 锚点引用格式

引用文档内具体章节时使用锚点，避免"见该文档"这种模糊描述：

```markdown
<!-- 正确 -->
详见 [DOC_SYNC.md § 4.4 大幅修改阈值](DOC_SYNC.md#44-大幅修改阈值)

<!-- 禁止 -->
详见 DOC_SYNC.md 中的相关说明
```

## 6. 文档生命周期

### 6.1 创建文档的触发条件

满足以下任一条件才允许新建文档：

- 新增了核心模块，现有 `ARCHITECTURE.md` + `design-docs/` 无法容纳其设计说明
- 引入了新的跨模块协作流程，无法归入任何现有 design-doc
- 新增了强制性约束或门禁，需要独立文档定义规则

**不允许**因"内容太多塞不下"而拆分文档——文档过长通常是内容准入失控的信号，应先删减。

### 6.2 更新文档的原则

文档更新**由漂移驱动**，不由"完善欲望"驱动：

- 代码变更 → 检查是否产生漂移 → 有漂移才更新
- 不允许在无代码变更的情况下主动"丰富"文档内容

### 6.3 废弃与删除

当某模块被整体删除或重构后，对应文档需同步处理：

1. 确认无其他文档引用该文档
2. 从 `DOC_SYNC.md` 的文档映射表中移除对应条目
3. 删除文件，在同一 commit 中完成

## 7. Agent 执行规范

本节定义 agent 在创建或修改文档时必须遵守的行为约束。

### 7.1 执行前必做

1. 读取本文档（`DOC_CONVENTIONS.md`）——内容约束全程强制执行
2. 读取 `DOC_SYNC.md`——确认当前操作的触发条件与受影响文档范围
3. 通过 `git diff` 理解**变更语义**，聚焦接口 / 状态机 / 流程变化，忽略实现细节

### 7.2 绝对禁止项

- 因"文档可以更完整"而主动增加内容
- 将纯重构（行为不变）标记为漂移并触发修改
- 无漂移时输出 `UPDATED`（应输出 `NO_CHANGES`）
- 生成"可能需要更新"、"建议考虑"类模糊建议——要么有明确 file:line 的具体建议，要么 `NO_CHANGES`
- 复制其他文档内容而非引用

### 7.3 判断优先级

遇到规则冲突时，优先级从高到低：

```
本文档内容约束（§ 2, § 4）
  > DOC_SYNC.md 漂移检测 checklist（§ 3）
    > 单个文档的历史风格
```

### 7.4 豁免情况处理

以下情况不修改文档，在报告中用一句话说明豁免原因：

- 纯重构（逻辑行为不变，仅代码结构调整）
- 变更仅在 `testdata/`、`*_generated.go`、`*_test.go`
- 文档已包含该信息
- 无法确定修改方向——标注 `UNCERTAIN`，给出分析，不猜测

## 8. 快速检查清单

提交文档变更前对照此清单：

```
内容准入（按职能类型）
  [ ] inform 类：内容属于架构决策 / 接口约束 / 流程全景，未写代码可直接读取的细节
  [ ] constrain 类：每条规则可执行可验证，禁止项说明了检测方式或违反后果
  [ ] verify 类：所有数值有来源（对应配置常量），无"大约"等模糊量词
  [ ] feedback 类：checklist 项可判断真假，流程描述与实际脚本一致

通用准入
  [ ] 无跨文档重复块（已改为链接引用）
  [ ] 无假设性内容（"可能"、"将来考虑"）
  [ ] 无纯实现细节

修改质量
  [ ] 满足修改前三问（事实错误 + 不可替代 + 等长或更短）
  [ ] 每处修改单独满足写入标准（没有它会做出错误决策？）
  [ ] 涉及架构图时，报告中标注 DIAGRAM_UPDATED 并附说明

结构格式
  [ ] 文档头部有一句话定位说明及职能类型声明
  [ ] 标题层级未跳级，最多 4 级
  [ ] 列表嵌套不超过 3 层
  [ ] 代码块有语言标识符

引用规范
  [ ] 引用其他文档时使用了具体的章节锚点链接
  [ ] 无内容的本地副本（直接引用 SSOT 来源）

生命周期
  [ ] 若新建文档，已更新 DOC_SYNC.md 的文档映射表
  [ ] 若删除文档，已同步清理所有引用该文档的链接
```
